# Troubleshooting & Fix Plan

The error `Microsoft.EntityFrameworkCore.Storage.RelationalConnection.OpenAsync` suggests a database connection issue. In the context of Render.com (free tier) and the code analysis, this is likely caused by **connection exhaustion** or **timeouts** due to inefficient querying.

The current implementation of `GetDashboardMetricsHandler` fetches **all** tasks from the database into memory (`await _context.ProjectTasks.ToListAsync()`) before calculating metrics. This is highly inefficient and can crash the application on constrained environments like Render's free tier.

## Root Cause Analysis
1.  **Inefficient Querying**: `_context.ProjectTasks.ToListAsync()` pulls the entire table.
2.  **Memory Pressure**: Loading all objects into memory spikes RAM usage, potentially hitting container limits.
3.  **Connection Timeouts**: Transferring large datasets takes time, keeping connections open longer.

## Proposed Fix: Optimize Dashboard Queries
We will refactor `GetDashboardMetricsHandler.cs` to perform aggregations (Count, Sum, Average) directly in the database using SQL, rather than in memory.

### Steps:
1.  **Use `AsNoTracking()`**: Disable change tracking for read-only queries to improve performance.
2.  **Database-side Aggregation**:
    - Replace `tasks.Count` with `await query.CountAsync()`.
    - Replace `tasks.Where(...).Sum(...)` with `await query.Where(...).SumAsync(...)`.
3.  **Efficient Data Retrieval**:
    - For lists like `RecentActivities`, `CurrentProjects`, and `UpcomingDeadlines`, use `.Take(N)` **before** `ToListAsync()` to fetch only the required rows.

### Example Change:
**Current (Bad):**
```csharp
var tasks = await _context.ProjectTasks.ToListAsync(); // Fetches 1000 rows
var count = tasks.Count;
```

**New (Good):**
```csharp
var count = await _context.ProjectTasks.CountAsync(); // Fetches 1 integer
```

This change will drastically reduce the data transferred and memory used, preventing the connection timeouts you are seeing on Render.
