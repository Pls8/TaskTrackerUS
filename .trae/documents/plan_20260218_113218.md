# Connection String Parsing Fix Plan

The error `Invalid port: -1` confirms that the parsing logic for the connection string is failing. Specifically, `new Uri(connectionString)` isn't correctly identifying the port, likely because the connection string from NeonDB might look slightly different than a standard URI (e.g., missing port or using a different scheme), or simply because the `Uri` class defaults the port to -1 if not explicitly present in the string.

Also, the user mentioned that NeonDB offers a direct .NET/EF Core connection string format. Using that directly is much safer than trying to parse a URI manually.

## Root Cause
1.  **Manual Parsing Failure**: The code `var databaseUri = new Uri(connectionString)` fails to extract the port correctly (`-1`).
2.  **Double Handling**: If the user (or Render) provides a valid ADO.NET connection string (Key=Value), my parsing logic might be corrupting it by trying to treat it as a URI.

## Solution
We will simplify the logic in `DependencyInjection.cs`.
1.  **Direct Use**: If the connection string *doesn't* start with `postgres://`, assume it's already a valid ADO.NET string (Key=Value) and use it as is. This allows the user to simply copy the ".NET" connection string from NeonDB and paste it into Render's environment variable.
2.  **Robust URI Parsing**: If it *is* a URI (starts with `postgres://`), we'll handle the default port (5432) if `Uri.Port` returns -1.

### 1. Update `DependencyInjection.cs`
I will modify the parsing block to:
- Check for `postgres://` prefix.
- If present, parse. If port is -1, default to 5432.
- If NOT present, use the string as-is.

### 2. User Instruction
I will advise the user to simply copy the **ADO.NET / Entity Framework** connection string from NeonDB (which looks like `Host=...;Database=...;`) and set that as the `DATABASE_URL` in Render. This bypasses the fragile URI parsing entirely.

I will start by making the code more robust, then provide the instruction.
