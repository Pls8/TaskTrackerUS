# Deployment Fix Plan: Render Database Connection

The error `System.ArgumentException: Couldn't set postgresql://...` indicates that the **Npgsql** driver in your .NET application is failing to parse the connection string provided by Render.

Render provides a connection string in the format: `postgresql://user:password@host/dbname`. However, Npgsql expects a standard ADO.NET connection string format like: `Host=host;Database=dbname;Username=user;Password=password`.

## Root Cause
- **Format Mismatch**: Render's environment variable `DATABASE_URL` (or similar) is a URI, but .NET's `Npgsql` driver historically required a keyword-value pair string.
- **Recent Npgsql Versions**: Newer versions *can* handle URIs, but sometimes configuration or specific driver versions cause issues. Or, the application code might be trying to modify the string incorrectly.

## Solution
We need to update the application startup logic (`Program.cs` or `DependencyInjection.cs`) to parse the Render-style connection URL and convert it into a format that Npgsql accepts.

### 1. Update `DependencyInjection.cs` (Infrastructure Layer)
I will modify how the `DbContext` is configured. We will add logic to detect if the connection string starts with `postgres://` or `postgresql://` and parse it manually.

**Proposed Logic:**
1.  Check if the connection string is a URI.
2.  If yes, extract `Host`, `Port`, `Username`, `Password`, and `Database`.
3.  Reconstruct the string as `Host=...;Port=...;Database=...;Username=...;Password=...`.

### 2. Verify `Program.cs`
Ensure that the connection string is being read from the environment variable correctly. Render usually sets a `DATABASE_URL` variable.

### 3. Rebuild and Push
After patching the code, we will rebuild the Docker image and push it to Docker Hub again.

I will start by examining `TaskTracker.Infrastructure/DependencyInjection.cs` to apply the fix.
